# 可选设备、多片段与六十进制时间界面设计

## 1. 背景

当前程序一次转换会同时生成 iPhone、标准 Android、vivo/iQOO 和 Windows 四套成品。界面只能设置一个片段，时间输入框直接显示十进制秒数。Windows 使用深色系统主题时，Qt 消息框可能使用深色背景，却继承主窗口的深蓝色文字，导致完成提示难以阅读。

本次改造需要同时解决以下问题：

- 用户可以选择一个或多个目标设备，成品目录不再包含未选择设备的文件；
- 开始时刻、片段时长和封面时刻使用六十进制时间显示；
- 一个视频可以添加多个片段，每个片段分别生成一套独立成品；
- 信息、警告和错误弹窗在 Windows 深色与浅色主题下均清晰可读；
- 修改只在本地完成和测试，不推送 GitHub。

## 2. 目标与非目标

### 2.1 目标

1. 支持 `iphone`、`android`、`vivo`、`windows` 四种输出目标的任意非空组合。
2. 最终成品目录只发布所选目标需要的媒体文件、使用说明和 manifest。
3. 多个片段依次转换，每个片段拥有独立参数、独立目录和独立 manifest。
4. 所有时间控件以 `MM:SS.cc` 或 `HH:MM:SS.cc` 显示，同时继续以秒为内部数据单位。
5. 保持现有 FFmpeg、平台封装和原子目录发布能力。
6. 为新增行为和弹窗缺陷提供自动化回归测试，并运行真实 FFmpeg 冒烟测试。

### 2.2 非目标

- 不把多个片段拼接为一张 Live Photo。
- 不在本次改造中增加手机直传功能。
- 不改变各厂商动态照片的底层封装格式。
- 不支持同时处理多个输入视频。
- 不取消临时共用媒体文件；它们仍可在转换期间存在，但不得出现在未选择相应用途的最终成品中。
- 不在完成本地测试后自动提交或推送实现代码。

## 3. 方案选择

### 3.1 多片段界面

采用“共享编辑区 + 片段列表”。现有三个时间输入框和两个滑块继续作为当前片段的编辑器；其下增加片段列表、添加按钮和删除按钮。切换片段时，先保存当前编辑值，再加载目标片段。

未采用“每个片段一张完整卡片”，因为片段增多后页面和滑块数量会快速膨胀。未采用弹窗编辑，因为用户需要频繁对照视频总时长并微调参数。

### 3.2 设备输出流水线

采用“共享中间媒体 + 按目标选择性封装”。转换器只转码一次共用 MP4、提取一次共用 JPEG；随后仅运行所选平台的封装和校验。共用文件位于临时目录：

- 选择 Windows 时，将共用 JPEG 和 MP4 作为 Windows 成品发布；
- 选择 iPhone 时，生成 Apple JPEG 和 MOV；
- 选择标准 Android 时，生成单文件 Motion Photo；
- 选择 vivo/iQOO 时，生成同名 JPG 和 MP4 配对。

未采用“先生成全部再删除”，因为这仍会浪费封装和校验时间，也无法满足不生成未选文件的真实意图。未采用“每个平台独立完整转码”，因为同时选择多个平台时会重复转码。

### 3.3 弹窗样式

继续使用 `QMessageBox`，但为消息框、正文标签和按钮设置完整的显式样式。这样保留 Qt 的键盘、焦点和无障碍行为，同时消除系统深色背景与应用深色文字混用的问题。无需引入自定义弹窗类。

## 4. 数据模型

### 4.1 输出目标

在 `livephoto/core/models.py` 中定义：

```python
OUTPUT_TARGETS = frozenset({"iphone", "android", "vivo", "windows"})
```

`ConversionOptions` 新增不可变字段：

```python
targets: frozenset[str] = frozenset({"iphone", "android", "vivo", "windows"})
segment_label: str = ""
```

核心模型保留“全部设备”为默认值，以避免现有 CLI 和第三方 Python 调用在升级后静默改变行为；GUI 会明确传入用户勾选的目标。验证规则要求 `targets` 非空并且全部属于 `OUTPUT_TARGETS`。

`segment_label` 用于多片段目录命名和日志，例如 `片段01`。单片段 CLI 不传此字段时保持原目录命名。

### 4.2 界面片段模型

新增 `ClipSegment` 不可变数据类：

```python
@dataclass(frozen=True, slots=True)
class ClipSegment:
    start_time: float
    duration: float
    cover_time: float
```

它只描述单个片段的时间范围，不包含全局画质、静音、输入目录和输出目标。`validate(source_duration)` 复用与 `ConversionOptions` 相同的时间边界规则。

主窗口持有 `list[ClipSegment]` 和当前索引。共享编辑器中的临时值在切换、添加、删除和开始转换前写回列表。

### 4.3 选择性输出结果

现有 `OutputBundle` 的七个媒体路径全部必填，不适合选择性输出。改为：

```python
@dataclass(frozen=True, slots=True)
class OutputFile:
    role: str
    path: Path

@dataclass(frozen=True, slots=True)
class OutputBundle:
    directory: Path
    outputs: tuple[OutputFile, ...]
    manifest: Path
    instructions: Path
```

`OutputBundle.files` 返回所有 `outputs` 中的路径，再附加 manifest 和使用说明。调用方通过 `by_role(role)` 获取特定文件；角色不存在时返回 `None`。manifest 的 `files` 只记录实际发布的文件。

这是一次有边界的模型迁移：测试、验证脚本和 UI 需要改用角色查询，底层 Apple/Android/vivo 封装函数不变。

## 5. 六十进制时间控件

新增 `livephoto/ui/time_spinbox.py`，定义 `TimeSpinBox(QDoubleSpinBox)`。

### 5.1 显示规则

- 小于一小时：`MM:SS.cc`，例如 `00:03.00`、`12:05.40`；
- 一小时及以上：`HH:MM:SS.cc`，例如 `01:02:03.50`；
- `cc` 为百分之一秒；
- 控件内部 `value()` 始终返回总秒数浮点数；
- 单步调整保持 0.1 秒；
- 不再显示“秒”后缀，输入框旁的标签说明格式为“分:秒”。

### 5.2 输入规则

接受以下用户输入：

- `MM:SS`；
- `MM:SS.cc`；
- `HH:MM:SS`；
- `HH:MM:SS.cc`；
- 纯数字秒数，兼容用户粘贴旧值。

分钟和秒字段按六十进制解析。带小时格式中分钟与秒必须小于 60；不带小时格式允许分钟大于 59。非法内容由 Qt 验证状态阻止提交，不抛出未处理异常。

### 5.3 精度

界面显示两位小数，滑块仍以毫秒为整数单位。写入模型时使用控件的秒数值；FFmpeg 命令继续使用六位小数格式，不降低处理精度。

## 6. 多片段交互

### 6.1 初始化

视频探测成功后创建一个默认片段：开始 `00:00.00`、时长为 `min(3.0, 视频时长)`、封面为片段中点。项目仍只接受至少 1 秒的视频片段；不足 1 秒的视频显示无法创建有效片段的错误。

### 6.2 片段列表

在时间编辑区下方增加 `QListWidget`，每项显示：

```text
片段 1　00:00.00 → 00:03.00　封面 00:01.50
```

列表只负责选择和摘要；详细修改仍在上方共享编辑器完成。

### 6.3 添加

点击“添加片段”时：

1. 保存当前片段；
2. 新片段默认从当前片段结束位置开始；
3. 默认时长为 3 秒，但会缩短到视频剩余时长；
4. 如果剩余时长不足 1 秒，则从视频开头创建一个默认 3 秒片段，便于用户手动调整，而不是创建无效片段；
5. 封面默认位于新片段中点；
6. 新片段加入列表并立即选中。

允许片段之间重叠，因为用户可能希望从相近位置生成不同封面或不同时长的 Live 图。

### 6.4 删除

- 至少保留一个片段；只有一个片段时禁用删除按钮；
- 删除当前片段后优先选中其后继项，没有后继项则选中前一项；
- 删除后重新编号，成品目录编号以开始转换时的最终顺序为准。

### 6.5 切换和校验

切换列表项之前保存当前编辑器值。开始转换前依次校验所有片段；若第 N 个无效，选中它并显示“片段 N：具体原因”，不启动后台任务。

## 7. 设备选择交互

新增独立步骤卡“选择兼容设备”，位于片段卡与保存位置之间。四个复选框默认均不选中，旁边显示对应输出说明。

转换按钮启用条件增加“至少选择一种设备”。用户没有选择目标时，状态提示显示“请至少选择一种兼容设备”。忙碌期间设备复选框全部禁用。

按钮文字根据状态显示：

- 单片段：`生成所选设备的 Live 图`；
- 多片段：`生成 N 个片段的 Live 图`。

窗口副标题改为“按需生成 iPhone、Android、vivo/iQOO 或 Windows 动态照片文件”，避免继续暗示一次生成全部格式。

## 8. 转换流水线

### 8.1 单片段选择性转换

`Converter.convert()` 继续负责一个 `ConversionOptions`。执行顺序：

1. 探测和验证视频；
2. 创建临时目录；
3. 生成一次共用 H.264/AAC MP4；
4. 提取一次共用 JPEG；
5. 按 `targets` 条件生成 Apple、Android、vivo 输出；
6. 按所选目标运行条件校验；
7. 仅为所选目标生成使用说明章节；
8. 生成只包含实际文件的 manifest；
9. 原子发布成品目录；
10. 清理不需要发布的共用临时文件。

选择 iPhone 时，Apple MOV 可以直接复用共用 MP4 的媒体轨封装流程所需源文件；若现有 Apple 写入器要求 MOV 输入，则只在临时目录创建该 MOV。它不作为 Windows 文件发布。

进度采用阶段权重动态归一化：无论选择一个还是四个平台，最后都从 0 单调增长至 100，不为未选阶段保留空白进度区间。

### 8.2 批量片段转换

新增 `BatchConversionWorker`，接收同一 `Converter` 和 `tuple[ConversionOptions, ...]`，按顺序调用单片段 `convert()`。

批量进度换算：

```text
总进度 = ((已完成片段数 + 当前片段进度 / 100) / 片段总数) × 100
```

日志前缀显示 `[片段 2/4]`。全部完成后发出 `tuple[OutputBundle, ...]`。完成弹窗说明生成数量和总保存目录。

取消事件在所有片段间共享。当前 FFmpeg 终止后不再启动后续片段。已经原子发布的完整片段保留；正在处理的临时目录由转换器清理。

若片段失败，工作线程停止后续片段并发送带片段编号的错误信息。已完成片段保留，日志明确列出其目录。

## 9. 目录和文件命名

多片段 GUI 转换使用：

```text
原文件名_片段01_Live图_YYYYMMDD_HHMMSS
原文件名_片段02_Live图_YYYYMMDD_HHMMSS
```

同一秒内发生重名时继续使用现有 `_2`、`_3` 唯一后缀规则。片段标签必须经过 `safe_stem()` 清理。

不同目标的文件规则保持不变。仅选择 vivo/iQOO 时，成品目录包含 vivo 配对、manifest 和只含 vivo 说明的使用说明；其他组合依此类推。

## 10. manifest 与使用说明

manifest 升级为 `schema_version: 3`，新增：

```json
{
  "segment_label": "片段01",
  "targets": ["vivo"],
  "files": []
}
```

`targets` 使用固定顺序 `iphone`、`android`、`vivo`、`windows`，保证测试和版本差异稳定。`files` 只列出实际输出角色及使用说明，不记录已清理的内部中间文件。

使用说明由公共标题、所选设备章节、隐私说明组合而成。未选择平台的传输说明不出现，减少新手混淆。

## 11. 弹窗可读性

根因是 `APP_STYLE` 为所有 `QWidget` 设置深色文字，但没有为 `QMessageBox` 设置背景。Windows 深色主题提供深色原生消息框背景，于是形成深色文字叠加深色背景。

在 `APP_STYLE` 中显式增加：

```css
QMessageBox { background: #f4f7fb; }
QMessageBox QLabel { color: #17233c; background: transparent; }
QMessageBox QPushButton {
    background: #ffffff;
    color: #17233c;
    border: 1px solid #9db2ce;
}
```

同时设置按钮的 hover、pressed 和 focus 状态。完成、警告和错误消息框全部使用这一规则。测试验证样式规则存在；实际 Windows 检查通过程序化弹出消息框并截图确认。

## 12. 错误处理

- 未选择设备：转换按钮禁用，并显示可操作提示；
- 无有效片段：保持至少一个片段且禁止开始；
- 某片段越界：定位到该片段并在启动前提示；
- 某目标封装失败：删除当前片段临时目录，不发布半套文件；
- 多片段中途失败：保留此前完整目录，停止后续片段；
- 用户取消：不显示错误弹窗，状态和日志说明已完成数量；
- 打开成品：多片段完成后打开共同的保存根目录，而不是任一片段目录。

## 13. 测试策略

### 13.1 模型测试

- 接受合法目标集合；
- 拒绝空集合和未知目标；
- `ClipSegment` 接受合法时间范围并拒绝越界；
- `OutputBundle.by_role()` 正确处理存在和缺失角色。

### 13.2 时间控件测试

- 秒数格式化为 `MM:SS.cc`；
- 超过一小时格式化为 `HH:MM:SS.cc`；
- 解析两种冒号格式和纯秒数；
- 拒绝非法六十进制字段；
- `value()` 与显示文本往返保持百分之一秒精度。

### 13.3 流水线测试

- 单选每个平台时只产生对应角色；
- 多选时复用共用转码并产生目标并集；
- 未选平台的封装器和校验器不被调用；
- manifest 版本、targets 和 files 正确；
- 临时共用文件不泄漏到成品；
- 失败与取消继续满足原子清理保证。

### 13.4 UI 与工作线程测试

- 初始只有一个片段；
- 添加、切换、删除会保存正确参数；
- 至少保留一个片段；
- 没有目标时转换按钮禁用；
- 忙碌时片段和设备控件禁用；
- 批量工作线程按顺序执行并正确聚合进度；
- 批量取消后不启动剩余片段；
- 消息框样式包含明确的高对比度前景和背景。

### 13.5 集成与手工检查

1. 运行完整 pytest；
2. 运行 Markdown 链接和仓库内容检查；
3. 生成公开测试视频；
4. 分别执行 vivo 单目标、Windows 单目标和多目标真实转换；
5. 使用验证脚本检查实际成品；
6. 以 Qt offscreen 模式运行应用冒烟测试；
7. 在 Windows 当前主题下截图检查主界面与完成弹窗。

## 14. 兼容性和迁移

- Python 最低版本继续为 3.10；
- 不新增第三方运行依赖；
- CLI 默认仍生成全部目标，后续增加可重复 `--target` 参数以支持选择性输出；
- 旧 `ConversionOptions` 调用因 targets 有兼容默认值而继续工作；
- `OutputBundle` 属于内部公开程度较低的 Python API，但结构改变需要同步测试、CLI 和脚本；
- README、架构文档、兼容性文档和详细学习报告中“一次生成全部”的表述需要同步更新。

## 15. 验收标准

满足以下全部条件才算完成：

1. GUI 可选择一个或多个目标，未选目标不出现在最终成品和 manifest 中。
2. 三个时间输入框使用六十进制显示并能正常键入、调节和同步滑块。
3. 可以添加、切换和删除片段，每个片段独立生成目录。
4. 多片段进度、取消、失败和完成提示准确。
5. 完成弹窗在截图中具有清晰的浅色背景、深色文字和可读按钮。
6. 全部自动化测试、仓库检查、Markdown 链接检查和真实 FFmpeg 冒烟测试通过。
7. 本次实现没有推送到 GitHub。
